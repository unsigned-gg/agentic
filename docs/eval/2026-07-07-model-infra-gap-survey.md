**Recommendation.** vLLM Semantic Router, evaluated inside the already-pending Envoy AI Gateway v1.0 spike (D2) rather than as a separate track — it is an Envoy ExtProc filter, so the spike and this evaluation are one PoC in front of the existing LiteLLM data plane. Verification confirmed the project's health independently: Apache-2.0 at repo root, GOVERNANCE.md with multi-org maintainers (Red Hat + others), releases v0.1.0→v0.3.0 on a quarterly cadence, pushed 2026-07-07, no Ray dependency (code-search hits are incidental substrings), and Helm/operator deploy paths that satisfy the pinned-image rule (its root `install.sh` is a curl-bash quickstart to avoid — use the k8s path).

**What verification corrected (load-bearing).** The researcher's caveat — "pin it to header/model-name mutation only, verify passthrough bytes" — is **not implementable**. There is no byte-passthrough mode: `processor_req_body_anthropic.go` decodes inbound Anthropic requests to an OpenAI-shaped intermediate representation and rebuilds the body, and `anthropic/passthrough.go` documents that the translation silently drops `cache_control`, multi-block system prompts, and image fields, with best-effort/silent-drop replay of cache markers. The positive Anthropic-native path has no e2e coverage (explicitly out of scope in `anthropic_passthrough_regression.go`). Silent `cache_control` loss silently breaks Anthropic prompt caching — the hard constraint. Required mitigation: **exclude the Anthropic cache-sticky lane from the ExtProc filter entirely at the Envoy route level**; semantic-router routes only non-Anthropic lanes. Additionally, the reference `config.yaml` ships `semantic_cache: enabled: true` (Milvus, similarity thresholds 0.82/0.86) — a response cache that would shadow; it must be disabled in any house deployment. One PoC caveat carried forward: several classifier fine-tunes in the llm-semantic-router HF org lack license tags (base ModernBERT is Apache-2.0) — verify during the spike.

**Plano is refuted, not caveated.** The researcher recommended option (a): serve Arch-Router-1.5B on the 5090 behind the house's own router. Verification found this was never permissible. The current model LICENSE (HF commit 5b15689, 2026-04-02) is the "Katanemo Community License Agreement" administered by **DigitalOcean, LLC** (primary evidence of an acquisition), whose Section 4 requires a separate commercial license granted at DigitalOcean's sole discretion for any commercial use — and unsigned.gg is commercial infrastructure. The HF commit history shows the *prior* license (2025-11-16) already prohibited serving the model "as part of a framework, a proxy server, middleware, a gateway infrastructure product" except through archgw's own data plane — i.e., option (a) was prohibited under the old terms too, and the carve-out forces option (b). Option (b) — adopting the Plano proxy — is independently disqualified: its `hermesllm` crate handles Anthropic via request/response transform layers (`from_anthropic.rs`, `to_anthropic.rs`), not byte-preserving passthrough, violating the cache constraint and duplicating LiteLLM's role. Plano-Orchestrator-4B was swept into the identical license the same day. This is exactly the single-vendor relicensing failure mode that favors the vLLM-project-governed alternative. If router-as-a-model preference routing is specifically wanted, the honest path is an in-house fine-tune of an open-weights base on the Arch-Router paper's recipe (arXiv:2506.16655 — the paper is open even though the weights are not). Sakana Fugu, the frontier instance of the pattern, is hosted/proprietary and out of scope; an opaque third party in the Anthropic path would violate cache-stickiness regardless.

**Unresolved question for the thin-Go-router decision.** LiteLLM's built-in routing is load-balancing across deployments of the same model, not model choice, so the routing-brain gap is real. Whether it needs a learned brain at all — versus static role→model mapping from harness-declared roles — should be answered with data: run ulab LLMRouter's method zoo offline against unsigned-gg/bench traces before committing. Decide replace-vs-embed after the D2 spike.

**Recommendation.** The value in this lane is entirely in the KV/prefix sublane — 39k-token shared harness prompts are the canonical prefix-reuse workload. First action is free: verify vLLM automatic prefix caching is active and enable the native CPU KV-offload connector on the 5090 instance. Verification confirmed from source at tag v0.24.0 that `enable_prefix_caching: bool = True` is default-on (`vllm/config/cache.py:92`); note the official APC docs page still carries stale V0-era wording — confirm at runtime via `vllm:cache_query_hit_rate` metrics rather than trusting the docs. The offload path is native and mature: `OffloadingConnector` via `--kv-transfer-config`, with `CPUOffloadingSpec` (pinned host RAM) and `TieringOffloadingSpec` adding a filesystem/NVMe secondary tier (docs dated 2026-06-22; vLLM blog reports 2–22x TTFT reduction on offload hits, perf rework landed 0.12.0). **LMCache PoC stands, with corrections.** License is clean Apache-2.0 (LICENSE read directly; no CLA found), maintenance is strong (v0.5.1 published 2026-07-06 as claimed, same-day pushes, healthy contributor spread, 10.1k stars), no Ray in `requirements/common.txt`, and pinned Docker Hub images exist. Verification corrected the researcher's claim in three places: (a) **CPU-RAM-pinned KV does not survive engine restarts** — only the local-disk/remote backends persist; the original house_fit conflated tiers; (b) the **GDS backend is non-functional on WSL2** (GPUDirect Storage unsupported) — the NVMe tier must use the plain local-disk backend; (c) the RTX 5090 is Blackwell sm_120 — **use the cu129 image variants** (they exist: `lmcache/vllm-openai:v0.5.1-cu129`). Also noted: requirements pull google-cloud-bigtable/awscrt/redis unconditionally — a heavier surface than a pure connector, reinforcing containers-only deployment. One watch item, not a disqualifier: core maintainers run a commercial company (TensorMesh) — the license is clean today, but this lane just watched DigitalOcean relicense Katanemo's weights, so record it. The verifier's framing for the PoC: vLLM's native OffloadingConnector is the control baseline; LMCache earns its dependency weight only via disk persistence, restart survival, and eventual cross-node sharing (Hetzner phase, where Mooncake becomes the backend it drives).

**Semantic/response caches are structurally weak here.** The Anthropic constraint disqualifies any response cache from shadowing the agent lane, and agent coding traffic has near-zero semantic repeat rate even on local lanes — nothing in the sublane earns more than watch, and any future wiring must use an explicit per-model-group allowlist in the LiteLLM cache config scoped away from Anthropic.
